W3 Library Internals Overview

Library Internals

The WWW browsers and servers share a common architrecure, and a library of common code. (See also: Browser operation , and utility modules which are used throughout, and using the common library .).

In the contol flow diagram , common code is to the right of the grey line.

Application
This module is the main program, and is window-system-dependent. In the line mode browser , it is HTBrowse . In the WWW server, it is HTDaemon. The application is called by the operating system, and manages the overall running of the program. It asks the navigation module to load the default page.
Access Manager
The module which acually loads documents is based in HTAccess.c. This uses all the protocol modules . Given an anchor ID to jump to, it asks the anchor object for the address in order to load it.
History
This module records and replays on request the documents which the user vists.
Format manager
The format manager uses the parser modules to load the document as appropriate. It can also decide on the format of a file from its name.
Anchor object
The HTAnchor module takes care of creating anchors, managing the links between them and their attributes. This module is independent of the type of graphics object (text, line drawing etc). It stores hypertext addresses of anchors, and ensures that anchors with the same address are the same anchor. ( More on anchors )

Protocol modules

A protocol module is invoked by the navigation module in order to access a document. Each protocol module is responible for extracting information from a local file or remote server using a particular protocol. Depending on the protocol, the protocol module either builds a graphic object (e.g. hypertext) itself, or it passes a socket descriptor to the format manager for parsing by one of the parser modules.
File access
HTFile.c provides access to files, using HTFTP.c for remote access. The latter uses HTTCP for common TCP routines.
HTTP access
The HTTP module handles document search and retrieve using the HTTP protocol.
News access
The NNTP internet news protocol is handled by HTNews which builds a hypertext.
Gopher access
The internet gopher access to menus and flat files (and links to telnet nodes etc) is handled by HTGopher .
WAIS access
WAIS access is also implemented in a separate gateway program .

Format conversion modules

These modules allow different formats to be used to generate graphic objects. They invoked by the format manager . Currently we only parse HTML and plain text and MIME messages, but obviously other formats can be added.

See the format manager for the registration and typing of conversion modules.

HTML
Basic hypertext parsing is done by HTML.c which uses the simple SGML engine SGML.c as a basic tokeniser and element stack manager. This uses:
HTMLDTD
The table defining some aspects of the HTML document type description.
Plain text to HTML
Plain text can can be converted.

Streams

Streams are objects which accept characters (and strings and blocks). There is a basic stream object definition, and many individual modules with creation routines for different types of stream.

Streams can be thought of as like files open for write. Examples of specific stream objects are:

HTWriter
Writes to a socket or something opend with open().
File writer
Writes to a C FILE * object, as opened by fopen(), etc.
SGML parser
Parses a stream and generates a structured stream.
WAIS source parser
Parses a stream with a WAIS source file on it, and generates, via an HTML structured stream, a stream of the required type.

Structured streams

A structured stream is like a stream , but instead of just acepting text, it also accepts SGML events such as begina nd end elements. A structured stream therefore represents a structured document. The elements and entites are referred to by number, so each structured stream must be created with knowledge of the DTD which it is currently using.

The structured strams are definied in the SGML module .

Any instance of a structured stream has a related DTD which gives the rules and element anbd entity names for events on the structured stream. The only DTD which is currently in the library is the HTML DTD, in the HTMLDTD module.

The SGML parser uses a DTD to ouput to a structured stream from a stream of SGML. A hypertext editor will output to a structured stream when writing out a document. Many protocol modules output to a structures stream when generating their data structures.

Here are two structured stream modules:

HTML
This module is an important one as it actually PRESENTS a hypertext object to the user. GUI writers replcae this module with your own, or use it and replaced the HText module which it feeds.
HTMLGen
This structured stream regenerates a plain stream. It makes reference to the HTML DTD to regenerate the names of enetities and elements. This is used by the server to pass a hypertext document on to the client, and also by the client to save HTML.

Graphic objects

A graphic object is a (complex) displayable entity. It is built by a protocol module directly or using a parser. Graphic objects are in general necessarily coded differently on diferent window systems. The graphic object is resposible for displaying istelf, catching mouse clicks, and calling the navigation object in order to follow links. We use the more common term "document" to describe the logical entity which a graphics object represents and displays.
HText
This object is window-system dependent. In the line mode browser, the GridText module is the hypertext object, providing the generic functionality of this module.

General modules

System specifics
The tcp.h file includes system-specific include files and flags for I/O to network and disk.
Utilities
The HTUtil.h file includes things we need everywhere, generally macros for declarations, booleans, etc.
URL parsing
This module has various utilities fo handling W3 addresses.
Chunks
Chunks are variable length arrays, a sort of apology for real strings in C.
Strings
A few manipulation routines for strings of charcters.
