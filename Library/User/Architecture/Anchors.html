Anchors in the WWW architecture

Anchor Description

Anchors represent any references to graphic objects which may be the sources or destinations of hypertext links. Here follows a general description of their implementation in the WWW Library of Common Code. There are basically two types of anchors:
parent anchors
Represents whole graphic objects. That is, the target of a link pointing to a parent anchor is the full contents of the graphic object. A parent anchor is made the target of a link by making a HREF without any # appended to the URI.
child anchors
Represents subparts of a graphic object. A subpart is declared by making a NAME tag in the anchor declaration. A child anchor is the target of a link if the HREF link declaration contains a # appended to the URI.
The anchors are organized into a sub-web which represents the part of the web that the application (often through user interaction) has been in touch with. In this sub-web, any anchor can be the source of zero, one, or many links and it may be the target of zero, one, or many links. That is, any anchor can point to and be pointed to by any number of links. Having an anchor being the source of many links is often used in the POST method, where for example the same data object is to posted to a News group, a mailing list and a HTTP server.

Every request and hence every graphic object has a parent anchor associated with it. Anchors exist throughout the lifetime of the application, but as this generally is not the case for graphic objects, it is possible to have a parent anchor without a graphic object. If the data object is stored in the client cache, the parent anchor contains a link to it so that the client can access it through the Cache Manager.

Every parent anchor points to a remote data object. In the case of posting an anchor to a remote server, the data object is yet to be created.
